# Cache TVShowAllSeasonsWithDetails + drop search IMDb override

## Overview

Two related cleanups on the tvmeta layer.

**Part A — cache `*AllSeasonsWithDetails`.** Move the TMDB response cache one
level up: cache the fully-assembled `*AllSeasonsWithDetails` (post-override)
keyed by `(id, lang)`, replacing the per-season `episodesCache`. Today every
`/id/{id}` request — even a warm one served entirely from `episodesCache` —
still pays for:

- N deep-copies (`cloneSeasonEpisodes`) per request, one per season, allocating a
  fresh `*TVShowSeasonEpisodes` plus a `[]*TVShowEpisode` and N `*TVShowEpisode`
  structs.
- The full `overrideEpisodeRatings` pass: `seriesIMDbID` lookup + a linear walk
  over every episode calling `c.ratings.EpisodeRating` per episode.
- The errgroup fan-out plumbing, even when each child returns immediately from
  cache.

Caching the post-override result eliminates all three on warm hits — the
handler gets a shared, read-only `*AllSeasonsWithDetails` and just slices into
it. The TTL bounds staleness identically to the existing `detailsCache`.

**Part B — drop the IMDb series-level override on search.** Today
`SearchTVShows` runs an N-way TMDB external_ids fan-out (bounded at 8 parallel)
to resolve every result's IMDb ID, then calls `SeriesRating` per result to
overlay an IMDb rating on top of TMDB's `VoteAverage`. The cold-cache cost
(~3 batches × ~200ms for a 20-result page) dominates search latency. TMDB's
series-level vote average is good enough at series granularity; the IMDb
overlay's real value is *episode*-level (`/id/{id}`), where it stays.

Dropping the search override removes `overrideSeriesRatings`, `cloneTVShows`,
the external_ids fan-out from the search path, and the `SeriesRating` method
from the `RatingsProvider` interface (no other callers in `tvmeta`).
`SearchTVShows` collapses to a thin `searchCache.GetOrFetch` wrapper returning
a shared, read-only `*TVShows` — symmetric with the new `allSeasonsCache`.

**Goals:**
1. Skip the override loop and per-call clone allocations on warm `/id/{id}` hits.
2. Drop the now-redundant `episodesCache` (its sole caller is
   `TVShowAllSeasonsWithDetails`) and `cloneSeasonEpisodes` along with it.
3. Cut cold and warm `/search/{q}` latency by removing the IMDb series-rating
   overlay and its TMDB external_ids fan-out.
4. Strip the long shared-pointer / override-mutation comment blocks that were
   only there to defend the old layered designs in both code paths.

**Out of scope:**
- `detailsCache` stays. `TVShowDetails` is still called directly from
  `featured.go`'s extras refresh, so a top-level cache on
  `TVShowAllSeasonsWithDetails` does not replace it.
- The episode-level IMDb override in `/id/{id}` stays. Per-episode rating is
  where the IMDb dataset's granularity actually beats TMDB.
- `imdbratings.Provider.SeriesRating` (the concrete impl in the imdbratings
  package) stays. After this plan it has no production caller, but it's a
  separate package with its own test suite — pruning it is a hygiene PR for
  another day, called out in Post-Completion / Future work.
- No public API changes (`/id/{id}` and `/search/{q}` response shapes
  unchanged; the `Rating` field on search results just sources from TMDB
  instead of IMDb).

**Related patterns found:**

  changes. The Prometheus method label just gains `all_seasons` and loses
  `episodes`; `search` stays.
- `singleflight` collapses concurrent `/id/{id}?id=X&language=Y` hits; the
  fan-out and override only run once per warm key.
- `TVShowEpisodesBySeason` reverts to a thin pass-through (no cache wrap),
  matching the shape it had before the original cache plan.
- `SearchTVShows` collapses to the same shape: a single
  `searchCache.GetOrFetch` wrapping the raw TMDB call + parse + popularity
  sort.
- `imdbIDCache` (`sync.Map` on `Client`) keeps existing per-id TMDB
  external_ids resolutions. It's still used by the surviving
  `overrideEpisodeRatings` path; the search caller goes away with this plan.

**Non-obvious wrinkles:**
- The cached `*AllSeasonsWithDetails` already has IMDb overrides applied. The
  TTL must therefore bound IMDb-rating staleness, not just TMDB staleness.
  Since the IMDb provider refreshes every 24h, a 6h TTL on this cache means
  ratings can lag the freshest provider snapshot by up to 6h — an explicit
  trade-off this plan accepts in exchange for skipping the per-request
  override pass. Document this on the new config field.
- The cache fetcher must NOT inherit the caller's context-cancellation
  semantics across the fan-out. The `responseCache.GetOrFetch` already
  detaches via `context.WithoutCancel` — the fan-out inside the fetcher
  uses that detached context, so a cancelled caller doesn't poison the cache
  for other waiters. The current `errgroup` already passes `ctx` through;
  point is that the `ctx` it sees is the detached one from `GetOrFetch`, no
  extra plumbing needed.
- The cached value is shared. Returning it directly from
  `TVShowAllSeasonsWithDetails` is safe because the only caller
  (`by_id.go::idHandler`) reads it through `flattenSortedByRating`, which
  only *reads* fields and copies primitives into a fresh `[]episode`. No
  mutation. Verify this still holds before claiming the task done; if a
  future caller wants to mutate, they deep-copy themselves — same contract
  as `featuredExtras.view()`.
- `episodesCache` removal is a hard cut: no tier-fallback, no compatibility
  shim. `TVShowEpisodesBySeason` continues to exist (still called by the
  cache fetcher) but issues TMDB calls every time it is invoked — i.e. only
  on `allSeasonsCache` misses.
- The search rating change is user-visible in one narrow sense: the `Rating`
  field in `/search/{q}` results sources from TMDB instead of IMDb after
  this plan. Magnitudes are typically within ±0.3 across the catalog;
  episode-level granularity in `/id/{id}` is what users actually click
  through for, and that path is unchanged. Mention this in the README
  update so the JSON example's prose stays accurate.
- Removing `SeriesRating` from the `tvmeta.RatingsProvider` interface does
  not require touching `imdbratings.Provider.SeriesRating` — the concrete
  type just exposes one more method than the interface needs, which is
  fine. The dead-code prune is captured in Future work.

- **Testing approach**: Regular (code first, then tests). Same convention as
  the prior cache plan; this plan changes the cache shape, not the testing
  philosophy.
- Complete each task fully before moving to the next.
- Make small, focused changes.
- **CRITICAL: every task MUST include new/updated tests** for code changes in
  that task — write tests for new functions/methods, update tests for modified
  behavior, cover both success and error scenarios.
- **CRITICAL: all tests must pass before starting next task** — no exceptions.
- Run `make test-race ./internal/pkg/tvmeta/...` after every task that touches
  cache wiring or the fan-out, since the cache sits on the request hot path.

  `CacheConfig{}` (used by tests and by anyone who hasn't opted in) still
  yields a fully-disabled cache with deterministic per-call behavior.

## Technical Details

**New cache shape**

```go
// in client.go
type allSeasonsKey struct {
    id   int
    lang string
}

type Client struct {
    client           tmdbClient
    ratings          RatingsProvider
    imdbIDCache      sync.Map
    detailsCache     *responseCache[detailsKey, *TvShowDetails]
    allSeasonsCache  *responseCache[allSeasonsKey, *AllSeasonsWithDetails]
    searchCache      *responseCache[searchKey, *TVShows]
}
```

**New `TVShowAllSeasonsWithDetails`** (final shape, with aggressive comment cleanup):

```go
// TVShowAllSeasonsWithDetails returns the series details and per-season
// episode lists with IMDb rating overrides applied. The returned pointer is
// shared with concurrent readers and must be treated as read-only.
func (c *Client) TVShowAllSeasonsWithDetails(ctx context.Context, id int, language string) (*AllSeasonsWithDetails, error) {
    return c.allSeasonsCache.GetOrFetch(ctx, allSeasonsKey{id: id, lang: language},
        func(ctx context.Context) (*AllSeasonsWithDetails, error) {
            return c.buildAllSeasonsWithDetails(ctx, id, language)
        })
}

func (c *Client) buildAllSeasonsWithDetails(ctx context.Context, id int, language string) (*AllSeasonsWithDetails, error) {
    tvShowDetails, err := c.TVShowDetails(ctx, id, language)
    if err != nil {
        return nil, fmt.Errorf("get all episodes: %w", err)
    }

    eg := errgroup.Group{}
    seasons := make([]*TVShowSeasonEpisodes, tvShowDetails.SeasonsCnt)
    for i := 1; i <= tvShowDetails.SeasonsCnt; i++ {
        eg.Go(func() error {
            episodes, err := c.TVShowEpisodesBySeason(ctx, id, i, language)
            if err != nil {
                return fmt.Errorf("get all episodes for season %d: %w", i, err)
            }
            seasons[i-1] = episodes
            return nil
        })
    }

    if err := eg.Wait(); err != nil {
        return nil, err
    }

    c.overrideEpisodeRatings(ctx, id, seasons)

    return &AllSeasonsWithDetails{
        Seasons: seasons,
        Details: tvShowDetails,
    }, nil
}
```

Note: `TVShowEpisodesBySeason` now returns the canonical (un-shared) value
each call — it's no longer cached — so storing the returned pointer in
`seasons[i-1]` directly is safe. `overrideEpisodeRatings` then mutates those
freshly-allocated `*TVShowEpisode`s in place. The whole composite is what
gets cached.

**Defaults:**

| Cache | Size | TTL |
| --- | --- | --- |
| `details` | 1024 | 6h (unchanged) |
| `all_seasons` (new) | 1024 | 6h |
| `search` | 256 | 30m (unchanged) |

**Memory footprint estimate** (revise after Task 9 measurement):
- An `*AllSeasonsWithDetails` holds one `*TvShowDetails` (~500B) plus N
  `*TVShowSeasonEpisodes`, each ~2KB for an average 10–20 episode season.
  For a typical 5-season series: 500B + 5 × 2KB ≈ 10.5KB per entry.
- 1024 entries × ~10.5KB ≈ 10–11 MB resident — same order of magnitude as
  the dropped `episodesCache` (which capped at ~8MB), so net memory impact
  is roughly neutral.

**Why these TTLs:**
- All-seasons (6h): mirrors `details` TTL. The cached value carries a
  snapshot of IMDb-overridden ratings; with the IMDb provider on a 24h
  refresh cadence, 6h bounds rating-snapshot staleness to a quarter of a
  refresh window.

**Failure modes & mitigations:**
- TMDB error during fan-out: not cached (next call re-tries).
- IMDb provider not ready at fetch time: override is a no-op, cached value
  carries TMDB ratings; the entry remains valid until TTL expires, so a
  series fetched during provider-warmup may serve TMDB-only ratings for up
  to 6h. Acceptable: same envelope as the existing details cache and the
  search cache.
- Process restart: cache lost (intentional, in-memory only).
